# Sample atoms тАФ fixture data for backend development

Real atoms from Stoney's Meta ad-account, generated by `scripts/import_*.py` on
2026-05-25. They serve as a snapshot for Mubashir and the backend to develop
against without needing Meta credentials.

## Contents

- `performance/<ad_id>.json` тАФ 100 ad-performance atoms (90 days of history)
- `creative/<ad_id>.json` тАФ 100 ad-creative atoms with copy variants and image
  metadata. **Joinable 1:1 with `performance/` on `ad_id`** тАФ the canonical
  set is defined by `performance/` and the creatives were pulled via
  `import_meta_creative.py --ad-ids-from atoms/performance/`. See
  [`docs/notes/2026-05-25-fix-ad-id-join.md`](../../docs/notes/2026-05-25-fix-ad-id-join.md)
  for the bug this snapshot fixes (the previous snapshot only overlapped on
  5 ad_ids).
- `../images/<ad_id>_<n>.jpg` тАФ 103 image files covering ~75 of the 100
  creative atoms (~17 MB). The remaining ~25 ads are videos (`images: []` in
  the atom) тАФ video-frame extraction is a future addition.
- `../meta_export.csv` тАФ 20-row CSV with the same canonical `ad_id`s as
  `performance/`, sorted by spend descending. Schema kept compatible with
  `scripts/parse_meta_csv.py`. The `ROAS` column is blank because Meta alone
  does not produce ROAS (that requires a Shopify join, layer 4). `CTR` is in
  percent (Meta API convention) not fraction.

## Performance atom schema

Fields as delivered by the Meta Insights API at level=`ad`:

```
ad_id, ad_name, campaign_id, campaign_name, adset_id, adset_name,
impressions, clicks, ctr, cpc, cpm,
spend, reach, frequency,
date_start, date_stop
```

Numeric fields (`impressions`, `clicks`, `ctr`, `cpc`, `cpm`, `spend`, `reach`,
`frequency`) are stored as `float`; the rest as strings.

## Creative atom schema

```
ad_id, ad_name, created_time, date,
creative_id, creative_name, object_type, instagram_permalink_url,
headlines[], bodies[], descriptions[], ctas[], link_urls[],
images[]   // each: {ref, url, hash, width, height, description}
```

Notes:

- Stoney runs **dynamic-creative ads**: each ad has multiple text variants (typically 5 or 9 headlines and bodies) that Meta rotates per placement. Stored as arrays.
- `images[].ref` points to `images/<ad_id>_<n>.jpg` (relative to repo root). For the 30 committed atoms, the same filename also lives at `sample_data/images/<basename>` тАФ consumers running against the snapshot can resolve there. To regenerate the full 172-image set locally (27 MB), run `python scripts/import_meta_creative.py --limit 100` with your own Meta credentials.
- `images[].url` is a signed Meta CDN URL that **expires** (typically within days). For long-lived access, use the hash to re-resolve via `/act_<id>/adimages?hashes=[..]`.
- `images[].width` ├Ч `height` distinguishes feed (1080├Ч1080) from story (1080├Ч1920).
- `images[].description` is a **factual** vision-LLM description (architecture layer 3) of what is in the image тАФ subjects, setting, colors, visible text. Generated by `scripts/describe_creative_images.py` using Claude Sonnet vision. No marketing interpretation; "not visible" rather than guess. PLN can use this as a feature alongside copy text and numeric performance metrics.
- A small minority of ads are video тАФ those have `images: []`. Video-frame extraction is a future addition.

## Important

- **Not refreshable.** This is a snapshot. The live pipeline (production runtime)
  is coming later тАФ see the next data-runtime ADR (still to be written, together
  with Mubashir).
- **Not test fixtures.** Don't use these as test fixtures asserting exact values тАФ
  numbers age and don't update along with reality. Same for ad copy: dynamic
  creatives change as Stoney's marketers iterate.
- **Not production data.** Use only for local development. The real pipeline writes
  to `atoms/` (gitignored).
- **Ad copy is in Dutch.** Stoney's customer-facing language. The repo's working
  language is English (per ADR 0004) but customer-facing data is preserved as-is.
