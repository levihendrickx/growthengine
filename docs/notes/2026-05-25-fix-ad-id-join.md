# 2026-05-25 — Fix: Ad ID set-mismatch between creative and performance atoms

## Problem (as observed in the committed `sample_data/`)

`sample_data/atoms/creative/` and `sample_data/atoms/performance/` each
contain 100 atoms — but they overlap on only **5 ad_ids**. 95 % of the
performance rows have no creative, and 95 % of the creatives have no
performance. PLN can't learn anything from data that doesn't join.

The same bug will reproduce in any fresh pipeline run, because the two
import scripts pull from different endpoints with different filters:

| Script | Endpoint | Filter | Returns |
|---|---|---|---|
| `import_meta_api.py` | `/act_<id>/insights` | `time_range = last N days`, `level=ad` | Ads that **spent** in that window (typically older, actively-running ads) |
| `import_meta_creative.py` | `/act_<id>/ads` | none (newest-first pagination) | The newest `--limit` ads (typically recently-created ads, many not yet running) |

For an account with more than `--limit` ads, the two sets diverge. In
Stoney's account that divergence is near-total.

## Fix

`import_meta_creative.py` now accepts `--ad-ids-from <dir>`. When set,
the script reads ad_ids from `<dir>/<ad_id>.json` filenames and fetches
exactly those ads via the Meta batch endpoint
`https://graph.facebook.com/<v>/?ids=<id1>,<id2>,...` (up to 50 ads per
request, so 200 ads = ~4 API calls — well within the rate budget from
ADR 0003).

## Recommended workflow

Performance defines the canonical ad set. The creative pull follows.

```bash
# 1. Pull performance for the analysis window. This defines which ads exist.
python scripts/import_meta_api.py --days 90 --limit 200

# 2. Pull creatives ONLY for those ad_ids. Guaranteed join.
python scripts/import_meta_creative.py --ad-ids-from atoms/performance/
```

Why performance-first:

- Ads with spend are the unit of analysis for PLN. Ads without spend are
  invisible to the learning step regardless.
- An ad on Meta always has a creative (you can't run without one), so
  fetching creatives for known-spending ads will succeed in practice.
- The reverse (creative-first, then fetch performance for those ids) is
  also possible but yields creatives without performance, which is the
  uninteresting half of the dataset.

## Backward compatibility

`--ad-ids-from` is opt-in. Running `import_meta_creative.py` without it
preserves the old behaviour (`/act_<id>/ads`, newest first) for ad-hoc
exploration. The script's docstring now flags this behaviour as
non-joinable.

## Drift detection

When `--ad-ids-from` is set, the script compares requested ad_ids
against returned ad_ids and warns on the difference. Meta returns no
entry for ads that are deleted, archived, or in a different account —
those will surface as missing.

## Not addressed in this change

- `sample_data/meta_export.csv` still uses synthetic ad_ids
  (`ad_001`, `ad_002`) that do not match any real Meta ad_id. This is a
  separate ID mismatch (CSV path is the legacy `parse_meta_csv.py`
  prototype). Plan: once we regenerate the atoms with real joined IDs,
  rebuild `sample_data/meta_export.csv` from a subset of those
  performance atoms so the CSV path stays consistent.

- The committed `sample_data/` is **not** regenerated by this PR — that
  requires Meta credentials in `.env` and is a follow-up step.

## Follow-up to ship the fix end-to-end

1. Put `META_ACCESS_TOKEN`, `META_API_VERSION`, `META_AD_ACCOUNT_ID` in
   `.env`.
2. Re-run the two-step workflow above against the live account.
3. Run `scripts/describe_creative_images.py` to populate layer-3
   descriptions on the new creative atoms.
4. Copy the new `atoms/performance/`, `atoms/creative/`, `images/` and
   replace the corresponding `sample_data/` contents.
5. Rebuild `sample_data/meta_export.csv` from the new performance atoms
   (script TBD, or simply by-hand for a 20-row subset).
6. Commit + PR.
